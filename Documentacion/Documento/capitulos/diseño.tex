\section{Arquitectura (Modelo C4)}

El diseño arquitectónico de \textit{Graphito} se estructura utilizando el Modelo C4, el cual permite describir el software a través de múltiples niveles de abstracción. Esto proporciona diferentes perspectivas técnicas, desde la visión contextual de las interacciones externas hasta el nivel más profundo de componentes.

\subsection{Nivel 1: Contexto}

El diagrama de contexto ilustra cómo el sistema \textbf{GRAPHITO} se relaciona con sus actores externos. En el centro de la arquitectura, el sistema provee las capacidades de procesamiento y análisis. A su alrededor se ubican: el \textbf{Docente}, quien interactúa para gestionar ejercicios, cargar el código de los archivos a analizar y visualizar los reportes; y la \textbf{API de LLM Externo}, que es consultada por la plataforma para generar automáticamente soluciones sintéticas que formarán parte del marco de referencias.

\subsection{Nivel 2: Contenedores}

En este nivel de abstracción se detalla la arquitectura de micro-servicios y bases de datos que componen el sistema. \textit{Graphito} se separa funcionalmente en los siguientes contenedores principales:

\begin{itemize}
    \item \textbf{Capa de Cliente (Front-end):} Interfaz web interactiva en \textit{React} que consume servicios REST, facilitando el trabajo del docente.
    \item \textbf{Núcleo de Aplicación (Back-end):} Servicio principal implementado en \textit{FastAPI} (Python) encargado de orquestar el sistema, gestionar las peticiones a la IA y procesar la lógica general.
    \item \textbf{DB Relacional (PostgreSQL):} Persistencia estructurada para metadatos del usuario, problemas y registro histórico integral del análisis.
    \item \textbf{Vector DB (ChromaDB):} Persistencia hiperdimensional de los \textit{embeddings} que agilizan las búsquedas por proximidad semántica espacial en las referencias.
\end{itemize}

\subsection{Nivel 3: Componentes}

Haciendo un acercamiento al motor central de analítica del sistema (el contenedor Back-end de FastAPI), se ilustran los componentes internos que materializan la funcionalidad de doble canal para Graphito:

\begin{itemize}
    \item \textbf{Orquestador Inteligente (LangGraph):} Componente maestro de orquestación que evalúa iteraciones, fusiona características híbridas e interactúa cíclicamente para extender las referencias funcionales.
    \item \textbf{Módulo de Análisis Semántico:} Utiliza de forma profunda el modelo pre-entrenado \textit{GraphCodeBERT} para interpretar características lógicas estructurales a través del grafo de flujo de datos (DFG).
    \item \textbf{Módulo de Estilometría:} Implementa un clasificador \textit{CharCNN} mediante convoluciones 1D a nivel de caracteres, capaz de clasificar el origen (humano o sintético) al aprender la textura de redacción original y capturar singularidades de formato (ej. sangrías o estilo de llaves).
\end{itemize}

El nivel de código ofrece el máximo detalle referencial. Se subdivide en la vista estática, representada por las clases y patrones, y la vista dinámica, que muestra el paso a paso transaccional.

\subsubsection{Diagrama de Secuencia}

Para el flujo de eventos, el diagrama de secuencia dicta el actuar de los módulos durante un análisis estándar:

\begin{itemize}
    \item \textbf{Ejecución Paralela:} Tanto \textit{GraphCodeBERT} (para interpretar la lógica estructural a través de grafos) como \textit{CharCNN} (para captar la huella estilométrica) se ejecutan de manera concurrente con la finalidad de optimizar los tiempos de inferencia en el servidor. 
    \item \textbf{Consolidación e Indicadores:} Una vez concatenado el vector unificado mediante la Fusión de Características (\textit{Feature Fusion}), se calcula su similitud del coseno contra las referencias indexadas en ChromaDB y se retornan los indicadores visuales a la capa de abstracción del \textit{frontend} para la revisión del docente.
\end{itemize}

Este diagrama es fundamental para comprender cómo el sistema transita entre las fases de procesamiento paralelo y la fusión de características finales. En la Figura~\ref{fig:diagrama_estados} se ilustra el flujo de estados transaccionales.

A continuación, se describen las etapas clave representadas en el modelo:

\begin{itemize}
    \item \textbf{S1. Configuración del Problema:} Estado inicial donde el sistema se encuentra a la espera de los parámetros base, como el lenguaje de programación, las restricciones del ejercicio y la carga de soluciones de referencia.
    
    \item \textbf{S2. Carga de Entregas:} Transición en la que el sistema recibe los archivos fuente en formato \texttt{.c} o \texttt{.cpp}. En este punto, se valida la integridad de los archivos antes de proceder al motor de analítica.
    
    \item \textbf{S3. Preprocesamiento de Doble Canal:} Estado crítico donde el código es bifurcado. Aquí se ejecutan simultáneamente las tareas de limpieza para el análisis de lógica y la preservación de formato para el análisis de redacción.
    
    \item \textbf{S4. Análisis Semántico:} El sistema transita a este estado para la extracción de características lógicas mediante grafos. El resultado es un vector denso (\textit{embedding}) que encapsula la funcionalidad del código.
    
    \item \textbf{S5. Análisis Estilométrico:} En paralelo al estado anterior, el sistema procesa la textura del código para identificar rasgos de origen humano o sintético, generando una probabilidad porcentual de generación por IA.
    
    \item \textbf{S6. Fusión y Cálculo:} Una vez obtenidos ambos vectores, el sistema entra en una fase de consolidación. Se realiza el cálculo de la similitud del coseno contra las referencias y se unifican las métricas obtenidas.
    
    \item \textbf{S7. Reporte Generado:} Estado final donde los resultados se persisten en las bases de datos y se presentan al docente mediante indicadores visuales. Desde aquí, el sistema puede finalizar el proceso o retornar al estado S1 para una nueva evaluación.
\end{itemize}

Este marco de trabajo rige el flujo de datos desde su recolección hasta su despliegue y monitoreo en el entorno de producción, garantizando la trazabilidad y la reproducibilidad de los experimentos.

% ---------------------------------------------------------
\subsection{Adquisición de Datos}
% ---------------------------------------------------------

Esta fase garantiza la disponibilidad del insumo principal para el sistema dual. Se consolida un corpus balanceado a partir de dos fuentes: la recolección de entregas académicas reales en C/C++ (\textit{IEEE Programming Homework Dataset}) que conforman la clase humana, y la generación automatizada de variantes equivalentes utilizando Modelos de Lenguaje (LLMs) que conforman la clase sintética. 

Una vez balanceadas las clases, en esta misma fase se realiza la \textbf{Partición del Corpus} en los conjuntos de Entrenamiento (\textit{Train}), Validación (\textit{Val}) y Prueba (\textit{Test}). Realizar esta separación antes de cualquier procesamiento asegura que no exista fuga de información (\textit{data leakage}) entre las muestras, garantizando que los modelos se evalúen con datos matemáticamente aislados.

% ---------------------------------------------------------
\subsection{Preparación de Datos}
% ---------------------------------------------------------

Dado el enfoque de doble canal, los datos crudos sufren transformaciones divergentes dependiendo del modelo de destino:
\begin{itemize}
    \item \textbf{Canal Semántico:} El código es sometido a un proceso de limpieza y normalización que elimina ruido léxico. Posteriormente, se procesa para extraer la topología lógica, construyendo el Grafo de Flujo de Datos (DFG).
    \item \textbf{Canal Estilométrico:} Se preserva el formato original (incluyendo indentación y comentarios). El código se mapea a un espacio vectorial unidimensional (cuantización a nivel de caracteres), generando la matriz secuencial que la red convolucional utilizará para identificar patrones sintéticos.
\end{itemize}

% ---------------------------------------------------------
\subsection{Entrenamiento de Modelos}
% ---------------------------------------------------------

En esta etapa se define el comportamiento de aprendizaje de la arquitectura. Se implementaron dos estrategias diametralmente opuestas para optimizar recursos computacionales y maximizar la precisión:

\begin{itemize}
    \item \textbf{Extracción de Características (GraphCodeBERT):} Debido a que este modelo fundacional ya posee un preentrenamiento masivo en estructuras de código, se utiliza estrictamente en modo de extracción de características. No requiere ajuste de pesos, utilizando la representación densa (\textit{embedding}) directa para el análisis espacial.
    \item \textbf{Entrenamiento del Clasificador (CharCNN):} La red neuronal convolucional se entrena para la tarea de clasificación binaria. Para evitar que el modelo simplemente memorice los datos, se aplican estrategias de prevención de sobreajuste (\textit{overfitting}), tales como el procesamiento por lotes pequeños (\textit{mini-batches}), desactivación aleatoria de neuronas (\textit{dropout}) y el paro anticipado (\textit{early stopping}) basado en el conjunto de validación.
\end{itemize}

% ---------------------------------------------------------
\subsection{Evaluación de Modelos}
% ---------------------------------------------------------

Una vez finalizado el ajuste de la red CharCNN, se procede a la validación rigurosa utilizando exclusivamente el conjunto de pruebas (\textit{Test}). Esta fase corrobora que los modelos cumplan con los objetivos de desempeño base (RNF-03 y RNF-04). Se calculan las métricas de clasificación (Exactitud, Precisión, Recall y F1-Score) para garantizar la confiabilidad en la detección de IA. Por su parte, la calidad del espacio vectorial semántico se valida comprobando la coherencia en los valores de la similitud del coseno ante algoritmos lógicamente idénticos.

% ---------------------------------------------------------
\subsection{Despliegue de Modelos}
% ---------------------------------------------------------

Los modelos empaquetados y validados se integran al núcleo del sistema. Esta fase implica la carga de los pesos definitivos en el entorno del \textit{backend} (API REST). Aquí se codifica la \textbf{Fusión de Características (\textit{Feature Fusion})}, el algoritmo interno que concatena el vector semántico y la probabilidad estilométrica en una sola estructura analítica lista para ser consultada por el docente.

% ---------------------------------------------------------
\subsection{Monitorización de Modelos}
% ---------------------------------------------------------

La fase final asegura la estabilidad del sistema en el entorno operativo. Consiste en la ejecución de pruebas de integración continua diseñadas para verificar que el orquestador respeta las Reglas de Negocio establecidas, y que el rendimiento general del sistema (como los tiempos de latencia durante la inferencia híbrida) se mantiene dentro de los márgenes de los Requerimientos No Funcionales.

\section{Diseño de Base de Datos}
% ---------------------------------------------------------

Se ha implementado un sistema de persistencia híbrido que combina el almacenamiento relacional para datos estructurados y el almacenamiento vectorial para el manejo de representaciones densas (embeddings).

\subsection{Modelo Relacional (Diagrama Entidad-Relación)}

El repositorio relacional gestiona la persistencia de metadatos y registros históricos mediante el motor \textbf{PostgreSQL}. La arquitectura del modelo permite que cada \textbf{DOCENTE} administre de forma independiente sus entidades de \textbf{PROBLEMA}. A su vez, cada problema se vincula con múltiples instancias de \textbf{CODIGO\_FUENTE}, permitiendo diferenciar entre las entregas de los alumnos y las soluciones de referencia generadas o recolectadas.

La lógica central de la auditoría se consolida en la tabla \textbf{REPORTE\_ANALISIS}. Esta entidad posee una relación dual con el código fuente para contrastar una entrega específica frente a una referencia seleccionada, almacenando los índices de similitud lógica y probabilidad sintética resultantes. Finalmente, el sistema registra hallazgos técnicos en la entidad \textbf{INDICADOR\_INTEGRIDAD} cuando se detectan anomalías en la estructura lógica o patrones estilométricos que requieren la atención del docente.

\subsection{Esquema de Colección Vectorial (ChromaDB)}
% =========================================================

Para la gestión de las representaciones densas y la ejecución de búsquedas por proximidad semántica, el sistema implementa una capa de persistencia heurística mediante \textbf{ChromaDB}. A diferencia del modelo relacional, esta base de datos está optimizada para el almacenamiento de tensores y el cálculo de distancias en espacios de alta dimensionalidad.

Como se muestra en la Figura~\ref{fig:esquema_chroma}, el diseño de la colección se mantiene bajo una estructura plana (\textit{flat schema}) para minimizar la latencia durante la fase de recuperación de información.

\begin{figure}[H]
    \centering
    % Ajusta el ancho al ser un diagrama compacto
    \includegraphics[width=0.45\textwidth]{figuras/diagramavectorial.png} 
    \caption{Esquema conceptual de la colección vectorial en ChromaDB.}
    \label{fig:esquema_chroma}
\end{figure}

La colección \texttt{codigos\_referencia} se articula bajo tres pilares fundamentales que garantizan la integridad del análisis:

\begin{enumerate}
    \item \textbf{Identificador Único (UUID):} Se utiliza el mismo identificador generado en la base de datos relacional (\texttt{CODIGO\_FUENTE.id}) para actuar como puente lógico, permitiendo al orquestador reconstruir la información del alumno tras una coincidencia vectorial.
    
    \item \textbf{Embedding Híbrido (769 Dimensiones):} Es el componente central del sistema. Este vector concatena dos señales distintas: 
    \begin{itemize}
        \item \textbf{Canal Semántico (0-767):} Dimensiones que capturan la lógica y el flujo de datos del código fuente.
        \item \textbf{Canal Estilométrico (768):} Un valor escalar que representa la probabilidad de origen sintético, permitiendo que la búsqueda por similitud considere tanto la funcionalidad como la textura de la redacción.
    \end{itemize}

    \item \textbf{Metadatos JSON:} Almacenan atributos clave como el \texttt{problema\_id} y el \texttt{lenguaje}. Esto permite que el sistema realice un pre-filtrado lógico, asegurando que la búsqueda de vecinos más cercanos se ejecute únicamente sobre el conjunto de soluciones de referencia que pertenecen al mismo ejercicio académico.
\end{enumerate}

Esta arquitectura híbrida asegura que \textbf{Graphito} no solo sea preciso en la identificación de similitudes, sino también eficiente en términos computacionales al segmentar el espacio de búsqueda.
