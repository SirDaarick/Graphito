\chapter{Análisis del Sistema}

\section{Análisis de Requerimientos}

En esta sección se describen los requerimientos funcionales y no funcionales del sistema propuesto, los cuales establecen las capacidades, restricciones y criterios de desempeño que debe cumplir la solución desarrollada. Estos requerimientos sirven como base para el diseño, implementación y evaluación del sistema, asegurando la trazabilidad entre las necesidades del evaluador y la arquitectura final.

% =========================================================
\subsection{Requerimientos Funcionales}
% =========================================================

Los requerimientos funcionales definen las capacidades y procesos que el sistema debe implementar para cumplir con las funciones de análisis, gestión de información y evaluación del código en el contexto académico.

\begin{itemize}
    \item \textbf{RF-01:} El sistema debe permitir el registro e inicio de sesión de docentes mediante credenciales cifradas para garantizar un espacio de trabajo privado.
    
    \item \textbf{RF-02:} El sistema debe permitir al docente la creación y gestión de un listado de ejercicios o problemas de programación (Repositorio Personal).
    
    \item \textbf{RF-03:} El sistema debe permitir la carga de código fuente en formato .c, .cpp y .txt asociados a un ejercicio específico.
    
    \item \textbf{RF-04:} El sistema debe realizar la normalización adecuada del código, eliminando comentarios para la extracción de grafos y preservando las estructuras de estilo para el análisis de autoría.
    
    \item \textbf{RF-05:} El sistema debe generar representaciones basadas en Grafos de Flujo de Datos (DFG) y n-gramas de caracteres.
    
    \item \textbf{RF-06:} El sistema debe realizar el análisis de similitud semántica mediante el modelo GraphCodeBERT para la generación de vectores de características (embeddings), cuantificando la proximidad entre ellos utilizando la métrica de similitud del coseno.
    
    \item \textbf{RF-07:} El sistema debe realizar el análisis estilométrico mediante el modelo CharCNN para calcular la probabilidad de autoría.
    
    \item \textbf{RF-08:} El sistema debe solicitar y recibir múltiples versiones funcionalmente equivalentes del problema proporcionado por el evaluador a través de un modelo LLM externo (Trayectoria A1).
    
    \item \textbf{RF-09:} El sistema debe permitir el almacenamiento persistente de los embeddings y resultados en una base de datos vectorial junto con sus metadatos lógicos.
    
    \item \textbf{RF-10:} El sistema debe generar un reporte bimodal que integre el índice de similitud semántica y la probabilidad de autoría de forma visual.
    
    \item \textbf{RF-11:} El sistema debe emitir alertas de integridad de forma automática cuando se detecten discrepancias críticas entre la lógica y el estilo de la entrega (Trayectorias A4 y A5).
    
    \item \textbf{RF-12:} El sistema debe permitir la consulta histórica de reportes de análisis realizados previamente sin necesidad de re-procesar los archivos fuente.
\end{itemize}

% =========================================================
\subsection{Requerimientos No Funcionales}
% =========================================================

Los requerimientos no funcionales establecen las restricciones, atributos de calidad y condiciones de desempeño que debe cumplir el sistema propuesto durante su operación.

\begin{itemize}
    \item \textbf{RNF-01:} Seguridad de datos: El sistema debe cifrar las contraseñas de los usuarios y utilizar tokens de sesión (JWT) para proteger la comunicación entre el cliente y el servidor.
    
    \item \textbf{RNF-02:} Persistencia híbrida: El sistema debe contar con un esquema de base de datos dual (SQL para metadatos de usuario y Vectorial para embeddings de código).
    
    \item \textbf{RNF-03:} El modelo GraphCodeBERT debe alcanzar un F1-Score mayor a 0.85 en tareas de detección de clones.
    
    \item \textbf{RNF-04:} El modelo CharCNN debe alcanzar una precisión de autoría mayor al 0.8.
    
    \item \textbf{RNF-05:} El sistema debe operar bajo los estándares C11 y C++17 para garantizar compatibilidad con el currículo de ESCOM.
    
    \item \textbf{RNF-06:} El sistema debe mantener una tasa de falsos positivos menor al 10\% mediante el orquestador de métricas.
    
    \item \textbf{RNF-07:} El tiempo de procesamiento total por entrega no debe exceder los 20 segundos.
    
    \item \textbf{RNF-08:} El sistema debe ser compatible con navegadores basados en Chromium versión 110 o superior.
    
    \item \textbf{RNF-09:} La interfaz debe ser responsiva y mostrar indicadores de progreso en tiempo real durante la inferencia de los modelos.
    
    \item \textbf{RNF-10:} El sistema debe garantizar una disponibilidad del 99\% para su uso en periodos de evaluación.
    
    \item \textbf{RNF-11:} El sistema debe validar la integridad y tipo de los archivos en el cliente antes de ser enviados al backend.
\end{itemize}

% =========================================================
\subsection{Reglas de Negocio}
% =========================================================

Las reglas de negocio definen los criterios que guían el funcionamiento y uso ético del sistema dentro del entorno académico.

\begin{itemize}
    \item \textbf{RN-01:} Confidencialidad de los datos: Un docente solo podrá visualizar, editar y consultar los ejercicios y reportes generados bajo su propia cuenta de usuario.
    
    \item \textbf{RN-02:} El sistema se limita a programas de cursos de programación de ESCOM-IPN que sigan los estándares C11 y C++17.
    
    \item \textbf{RN-03:} Integridad de la referencia: Toda entrega debe ser comparada contra al menos una solución de referencia proporcionada por el docente para ser considerada válida.
    
    \item \textbf{RN-04:} La decisión final de evaluación es responsabilidad exclusiva del docente; el sistema actúa únicamente como una herramienta de apoyo a la auditoría académica.
    
    \item \textbf{RN-05:} Se permite el uso de múltiples soluciones generadas por LLM para ampliar el espectro de comparación lógica, siempre que el docente las valide inicialmente.
    
    \item \textbf{RN-06:} La autoría del código se infiere mediante la desviación del vector estilométrico respecto al estándar humano o sintético identificado por el canal B.
\end{itemize}

\section{Identificación y descripción de actores}
% =========================================================

En el contexto del sistema propuesto, los actores representan las entidades externas que interactúan con la plataforma y que participan en el flujo de análisis y evaluación del código fuente. La correcta identificación de estos actores permite definir los casos de uso, delimitar responsabilidades y establecer la interacción entre los usuarios y los módulos del sistema.

A partir del análisis de requerimientos y de la arquitectura funcional del sistema, se identifican los siguientes actores principales:

% ---------------------------------------------------------
\subsection{Evaluador (Docente)}
% ---------------------------------------------------------

El evaluador es el actor principal del sistema, ya que utiliza la plataforma como herramienta de apoyo en el proceso de evaluación académica. Este actor corresponde a profesores o instructores de cursos introductorios de programación.

Entre sus principales responsabilidades se encuentran:

\begin{itemize}
    \item Proporcionar la descripción del problema o práctica a evaluar.
    \item Definir restricciones técnicas asociadas al ejercicio (por ejemplo, uso de estructuras específicas o limitaciones en librerías).
    \item Cargar los programas desarrollados por los estudiantes.
    \item Seleccionar o generar fuentes de referencia para la comparación.
    \item Analizar los resultados generados por el sistema, incluyendo la similitud semántica y la probabilidad de autoría.
    \item Tomar la decisión final sobre la evaluación del código.
\end{itemize}

Es importante destacar que el sistema no reemplaza la decisión del docente, sino que actúa como una herramienta de apoyo proporcionando métricas e información relevante para facilitar el proceso de evaluación.

% ---------------------------------------------------------
\subsection{Estudiante}
% ---------------------------------------------------------

El estudiante es el actor que genera el código fuente que será evaluado por el sistema. Aunque no interactúa directamente con la plataforma en todos los escenarios, su participación es fundamental dentro del flujo del sistema.

Sus principales características son:

\begin{itemize}
    \item Desarrolla soluciones a problemas de programación en C/C++.
    \item Entrega el código fuente como resultado de una actividad académica.
    \item Puede utilizar diferentes enfoques, estructuras o herramientas (incluyendo IA generativa) para resolver el problema.
\end{itemize}

El sistema analiza indirectamente el trabajo del estudiante con el objetivo de evaluar la equivalencia lógica de su solución y detectar posibles patrones de generación automática.

% ---------------------------------------------------------
\subsection{Modelo de Inteligencia Artificial Externo (LLM)}
% ---------------------------------------------------------

Este actor corresponde a un sistema externo basado en modelos de lenguaje (LLM), el cual es utilizado para generar múltiples soluciones funcionalmente equivalentes a un problema dado.

Sus funciones dentro del sistema son:

\begin{itemize}
    \item Recibir la descripción del problema y las restricciones definidas por el evaluador.
    \item Generar soluciones alternativas que cumplen con dichas restricciones.
    \item Proporcionar un conjunto de referencia ampliado para el análisis de similitud semántica.
\end{itemize}

Este actor permite enriquecer el proceso de comparación, ya que introduce variabilidad en las soluciones de referencia, mejorando la capacidad del sistema para detectar equivalencias funcionales.

% ---------------------------------------------------------
\subsection{Sistema}
% ---------------------------------------------------------

El sistema actúa como el núcleo del proceso de análisis, integrando los diferentes módulos de procesamiento y modelos de inteligencia artificial.

Sus principales funciones incluyen:

\begin{itemize}
    \item Preprocesamiento y normalización del código fuente.
    \item Generación de representaciones estructurales (DFG y n-gramas).
    \item Cálculo de similitud semántica mediante GraphCodeBERT.
    \item Análisis estilométrico mediante CharCNN.
    \item Generación de reportes con métricas de similitud y probabilidad de autoría.
\end{itemize}

El sistema coordina la interacción entre los actores y centraliza el flujo de datos necesario para el análisis.

% ---------------------------------------------------------
\subsection{Base de Datos}
% ---------------------------------------------------------

La base de datos es un actor de soporte que permite el almacenamiento y recuperación de información relevante para el sistema.

Sus funciones principales son:

\begin{itemize}
    \item Almacenar embeddings generados por los modelos.
    \item Guardar metadatos asociados a los programas analizados.
    \item Permitir la reutilización de información para futuras comparaciones.
\end{itemize}

La existencia de este componente permite mejorar la eficiencia del sistema y facilitar análisis posteriores.

\section{Trayectorias principales y alternativas}
% =========================================================

Las trayectorias del sistema describen el flujo de interacción entre el evaluador y la plataforma Graphito, detallando las secuencias de acciones necesarias para llevar a cabo el análisis de código. Estas trayectorias permiten identificar tanto el comportamiento esperado del sistema como posibles escenarios alternativos ante condiciones específicas.

% ---------------------------------------------------------
\subsection{Trayectoria principal}
% ---------------------------------------------------------

La trayectoria principal describe el flujo estándar de uso del sistema para realizar el análisis de código en un contexto académico.

\textbf{Flujo principal:}

\begin{enumerate}
    \item El evaluador accede a la plataforma web del sistema.
    \item El evaluador define el problema de programación a evaluar, incluyendo restricciones y criterios relevantes.
    \item El evaluador carga los programas desarrollados por los estudiantes.
    \item El sistema recibe el código y realiza un proceso de preprocesamiento y normalización.
    \item El sistema genera representaciones estructurales del código, incluyendo grafos de flujo de datos y características estilométricas.
    \item El módulo de análisis semántico genera embeddings del código utilizando GraphCodeBERT.
    \item El módulo de análisis estilométrico procesa el código mediante el modelo CharCNN.
    \item El sistema calcula la similitud semántica entre programas utilizando similitud del coseno.
    \item El sistema estima la probabilidad de autoría basada en patrones estilométricos.
    \item El sistema integra los resultados mediante el orquestador y genera un reporte final.
    \item El evaluador visualiza los resultados del análisis.
    \item El evaluador toma una decisión informada sobre la evaluación del código.
\end{enumerate}

% ---------------------------------------------------------
\subsection{Trayectorias alternativas}
% ---------------------------------------------------------

Las trayectorias alternativas describen variaciones del flujo principal que pueden ocurrir debido a condiciones específicas durante la ejecución del sistema.

\textbf{A1. Generación de soluciones de referencia mediante IA}

\begin{enumerate}
    \item El evaluador solicita la generación de soluciones de referencia.
    \item El sistema envía la descripción del problema a un modelo de lenguaje externo.
    \item El modelo genera múltiples soluciones funcionalmente equivalentes.
    \item El sistema incorpora estas soluciones al conjunto de comparación.
    \item El flujo continúa a partir del paso 4 de la trayectoria principal.
\end{enumerate}

\textbf{A2. Código con errores de formato o sintaxis}

\begin{enumerate}
    \item El sistema detecta inconsistencias durante el preprocesamiento del código.
    \item Se aplica un proceso de limpieza y normalización.
    \item Si el código es recuperable, el sistema continúa con el análisis.
    \item En caso contrario, se notifica al evaluador sobre el error.
\end{enumerate}

\textbf{A3. Ausencia de datos suficientes para análisis estilométrico}

\begin{enumerate}
    \item El sistema identifica que el código no contiene suficientes características estilométricas.
    \item El módulo estilométrico reduce su peso en la evaluación.
    \item El sistema prioriza el análisis semántico.
    \item Se genera el reporte con esta consideración.
\end{enumerate}

\textbf{A4. Alta similitud semántica detectada}

\begin{enumerate}
    \item El sistema detecta una similitud elevada entre programas.
    \item Se marca el resultado como caso relevante.
    \item Se prioriza su visualización en el reporte.
    \item El evaluador realiza una revisión más detallada.
\end{enumerate}

\textbf{A5. Inconsistencia entre análisis semántico y estilométrico}

\begin{enumerate}
    \item El sistema detecta discrepancias entre los resultados de ambos módulos.
    \item El orquestador aplica reglas de decisión.
    \item Se genera una alerta en el reporte final.
    \item El evaluador analiza el caso manualmente.
\end{enumerate}

\subsection{Descripción de Actores}

Para el funcionamiento del sistema se han identificado dos actores principales:

\begin{itemize}
    \item \textbf{Docente / Evaluador:} Actor principal humano que interactúa con la plataforma para configurar problemas, gestionar entregas y analizar los reportes de similitud.
    \item \textbf{Modelo LLM Externo:} Actor de sistema que actúa como proveedor de servicios de generación de código. Es consultado de forma opcional para robustecer el conjunto de referencias (Trayectoria A1).
\end{itemize}

\subsection{Descripción de los Casos de Uso}

A continuación, se describen los módulos funcionales que integran la solución \textit{Graphito}, estructurados de acuerdo con su rol en el flujo de auditoría académica:

\begin{itemize}
    \item \textbf{Gestión de Acceso y Biblioteca (UC1, UC2, UC3):} Facilitan la administración de la sesión del docente y el acceso al centro de mando (Biblioteca). Este módulo centraliza los ejercicios históricos y las comparaciones realizadas, permitiendo un entorno de trabajo persistente.
    \item \textbf{Gestión de Referencias y Pre-procesamiento (UC4, UC5, UC6, UC11):} Permiten al evaluador establecer el marco de comparación. Incluye la carga de soluciones base (UC4) y su enriquecimiento mediante un motor de IA externo para generar descripciones y variantes sintéticas (UC5, UC6). De forma obligatoria (\textit{include}), el sistema realiza la generación de \textit{embeddings} semánticos (UC11) sobre estas referencias.
    \item \textbf{Ejecución de Comparaciones (UC7, UC12):} Constituye el núcleo operativo del sistema. Al agregar una entrega para comparar (UC7), el sistema ejecuta automáticamente (\textit{include}) el análisis bimodal (UC12), el cual integra el procesamiento semántico de flujo de datos y el análisis de autoría estilométrica.
    \item \textbf{Resultados y Reporteo (UC8, UC9, UC10):} Proporcionan la interfaz de auditoría donde se listan las comparaciones (UC8) y se visualizan los reportes de similitud detallados (UC9). Finalmente, se permite la exportación de resultados en formato PDF (UC10) para su integración en actas de evaluación.
\end{itemize}

\section{Modelado de Procesos de Negocio (BPMN)}

Para la orquestación de las capacidades de inteligencia artificial y la gestión de la persistencia híbrida, se diseñó un modelo de procesos utilizando la notación BPMN 2.0. Este modelo, ilustrado en la Figura \ref{fig:bpmn_proceso}, detalla la interacción entre el docente, el motor de \textit{backend} y los módulos de inferencia de IA.

El proceso se divide en tres áreas de responsabilidad (\textit{lanes}) claramente definidas:

\begin{itemize}
    \item \textbf{Usuario (Docente):} Responsable de la carga de insumos (código de referencia y entregas de alumnos) y la toma de decisiones basada en los reportes generados.
    \item \textbf{Backend:} Actúa como el orquestador del sistema, gestionando la validación de archivos, la comunicación con las bases de datos (relacional y vectorial) y la sincronización de los flujos de datos.
    \item \textbf{Módulo de IA:} Ejecuta las tareas de cómputo intensivo, incluyendo la generación de variantes sintéticas mediante LLMs, la extracción de características semánticas con \textit{GraphCodeBERT} y el análisis estilométrico mediante \textit{CharCNN}.
\end{itemize}

\subsection{Descripción del Flujo de Trabajo}

El ciclo operativo de \textit{Graphito} se desglosa en tres fases críticas:

\begin{enumerate}
    \item \textbf{Fase de Gestión de Referencias:} El sistema ofrece tres caminos de ejecución para establecer la base de comparación. Si el código de referencia es nuevo, el \textit{backend} verifica su existencia en la base de datos relacional; en caso de no existir, el módulo de IA genera variantes sintéticas para robustecer el espacio vectorial. Si la referencia ya existe, se recupera directamente su representación densa (\textit{embedding}) desde la base de datos vectorial (\textit{ChromaDB}), optimizando los tiempos de respuesta.
    
    \item \textbf{Fase de Procesamiento Bimodal:} Una vez cargado el código del alumno, el sistema realiza un pre-procesamiento y normalización en el \textit{backend}. Posteriormente, el flujo se bifurca mediante una compuerta paralela para realizar un análisis dual: una extracción de lógica estructural (semántica) y un análisis de patrones de escritura (estilometría). Esta fase garantiza que el sistema capture tanto la intención funcional como la huella digital del autor.
    
    \item \textbf{Fase de Fusión y Resultados:} Los vectores resultantes son integrados en un proceso de fusión de características. El sistema utiliza una compuerta de sincronización para asegurar que tanto el vector del alumno como el de referencia estén disponibles antes de calcular la \textit{Similitud del Coseno}. Finalmente, el módulo de IA interpreta los resultados numéricos para generar un reporte de integridad que se persiste en el historial y se presenta al docente.
\end{enumerate}

\section{Definición de métricas de evaluación}
% =========================================================

La evaluación del sistema propuesto requiere la definición de métricas que permitan cuantificar su desempeño en los distintos módulos que lo componen. En particular, se consideran dos componentes principales: el análisis de similitud semántica y el análisis estilométrico.

Cada uno de estos módulos requiere métricas específicas que permitan evaluar su precisión, confiabilidad y utilidad en el contexto de evaluación académica, permitiendo así validar la efectividad del sistema como herramienta de apoyo a la decisión docente.

% ---------------------------------------------------------
\subsection{Métricas para el análisis de similitud semántica}
% ---------------------------------------------------------

El módulo de análisis semántico tiene como objetivo cuantificar el grado de similitud lógica entre dos programas. Para ello, se utilizan representaciones vectoriales (embeddings) generadas mediante el modelo GraphCodeBERT.

La principal métrica utilizada es la similitud del coseno, la cual mide el ángulo entre dos vectores en un espacio de alta dimensión. Su valor se encuentra en el rango [-1,1], donde valores cercanos a 1 indican alta similitud entre los programas analizados.

Esta métrica resulta especialmente adecuada para comparar embeddings, ya que permite evaluar la cercanía semántica entre programas independientemente de su forma sintáctica.

Adicionalmente, se considera la correlación con evaluaciones humanas como un mecanismo de validación externa, con el fin de determinar si los resultados generados por el sistema son consistentes con el juicio del evaluador.

% ---------------------------------------------------------
\subsection{Métricas para el análisis estilométrico}
% ---------------------------------------------------------

El módulo estilométrico se plantea como un problema de clasificación binaria, en el cual se busca determinar si un fragmento de código fue escrito por un humano o generado mediante inteligencia artificial.

Para evaluar el desempeño de este módulo, se utilizan métricas estándar de clasificación supervisada:

\begin{itemize}
    \item \textbf{Accuracy}: mide la proporción de predicciones correctas respecto al total de muestras evaluadas.
    \item \textbf{Precision}: indica la proporción de predicciones positivas correctas respecto al total de predicciones positivas realizadas.
    \item \textbf{Recall}: mide la capacidad del modelo para identificar correctamente los casos positivos.
    \item \textbf{F1-score}: representa la media armónica entre precisión y recall, proporcionando una medida equilibrada del desempeño del modelo.
\end{itemize}

Estas métricas permiten evaluar el comportamiento del modelo en distintos escenarios, particularmente en presencia de posibles desbalances entre clases.

    \item \textbf{Baja}: el riesgo es poco probable o su impacto es limitado.

Estos niveles permiten priorizar los riesgos y enfocar los esfuerzos de mitigación en aquellos que representan una mayor amenaza para el sistema.

\subsection{Selección y Justificación del Conjunto de Datos}

Para el entrenamiento del clasificador de autoría propuesto se requieren dos tipos de datos: código fuente escrito por humanos en un contexto educativo introductorio, y código equivalente generado por modelos de inteligencia artificial. A continuación se describe y justifica cada fuente utilizada, así como la estructura cuantitativa del dataset resultante.

\subsubsection{Código humano: IEEE Programming Homework Dataset}

El componente de código humano proviene del \textit{Programming Homework Dataset} publicado por Ljubovic y Pajic (2020) en \cite{Ljubovic2020Dataset}. Este dataset contiene envíos reales de estudiantes en dos cursos introductorios de programación universitaria: el curso A en lenguaje C y el curso B en C++, con entre 16 y 22 asignaciones por curso. Su selección se justifica por tres razones principales.

En primer lugar, el origen de los datos es directamente relevante al dominio del problema. Los envíos provienen de estudiantes en cursos de fundamentos de programación, que es exactamente la población objetivo del clasificador propuesto. Esto garantiza que el modelo aprenda patrones estilométricos propios del código estudiantil introductorio y no de programadores expertos o código de producción.

En segundo lugar, el dataset cubre los tipos de problemas de interés: estructuras de control, manejo de arreglos, funciones, estructuras de datos básicas y manejo de archivos, todos representativos del currículo de un curso introductorio de C/C++.

En tercer lugar, respecto al idioma de los identificadores y comentarios, el análisis del código fuente contenido en el dataset confirmó que este se encuentra escrito en serbocroata con alfabeto latino. Dado que los comentarios constituyen un rasgo estilométrico relevante puesto que codifican información sobre la frecuencia, extensión y estilo de documentación del programador, estos se conservan íntegramente como parte del input de CharCNN. La compatibilidad con código escrito en otros idiomas de alfabeto latino, como el español o el inglés, se garantiza a nivel de distribución de caracteres, a diferencia de datasets en idiomas con alfabetos no latinos como el griego o el cirílico.
